# Tanko - Especificación del Sistema

## Visión General
Sistema B2B de wallet descentralizado de combustible con integración Stellar/Soroban y Trustless Work para Multi-Release Escrow.

## Arquitectura

### Frontend (Next.js)
- Aplicación con autenticación via wallet Freighter
- Selección de rol (Conductor/Jefe de Flota)
- Rutas protegidas que requieren wallet conectada

### Backend (Express/Node.js)
- API REST con Prisma + PostgreSQL
- Integración con Trustless Work para escrows
- Servicio de estadísticas dinámico

### Smart Contract (Soroban - tanko-registry)
El contrato `tanko-registry` maneja la whitelist y configuraciones de conductores:

```rust
# Tipos de datos
#[contracttype]
#[derive(Clone)]
pub enum DataKey {
    Admin,
    Drivers,           // Map<Address, bool> - whitelist de drivers
    GasStations,       // Map<Address, bool> - whitelist de gasolineras
    DriverConfigs,      // Map<Address, DriverConfig> - configuración por driver
}

#[contracttype]
#[derive(Clone)]
pub struct DriverConfig {
    pub escrow_limit: i128,      // Límite máximo de escrow en stroops (10^7)
    pub escrow_used: i128,      // Monto usado actualmente en stroops
    pub escrow_available: i128, // Monto disponible en stroops
    pub is_active: bool,        // Si el driver está activo
    pub registered_at: u64,     // Timestamp de registro
}

// Funciones existentes
pub fn init(env: Env, admin: Address)
pub fn add_driver(env: Env, admin: Address, driver: Address)
pub fn add_station(env: Env, admin: Address, station: Address)
pub fn verify_tx(env: Env, driver: Address, station: Address) -> bool
pub fn remove_driver(env: Env, admin: Address, driver: Address)
pub fn remove_station(env: Env, admin: Address, station: Address)

// Funciones NUEVAS a implementar
pub fn get_driver_stats(env: Env, driver: Address) -> DriverConfig
// Retorna la configuración del driver. Si no existe, retorna DriverConfig con valores en cero.

pub fn update_driver_limit(env: Env, admin: Address, driver: Address, new_limit: i128)
// Solo el admin puede actualizar el límite de escrow de un driver.

pub fn record_usage(env: Env, admin: Address, driver: Address, amount: i128)
// Registra uso de fondos. Decrementa escrow_available, incrementa escrow_used.

pub fn reset_driver_usage(env: Env, admin: Address, driver: Address)
// Resetea el uso del driver (cuando se reembolsa o renueva el período).
```

### Flujo de Datos del Driver

1. **Frontend solicita balance** → `/api/v1/driver/stats/:address`
2. **Backend consulta Soroban** → `tanko-registry.get_driver_stats(driver)`
3. **Backend consulta Trustless Work** → balances de escrow activos
4. **Frontend muestra datos combinados**

## API Endpoints

### Driver Stats
```
GET /api/v1/driver/stats/:address
Response: {
  success: true,
  data: {
    escrowLimit: number,      // Límite desde Soroban (en USD con decimals)
    escrowUsed: number,       // Usado desde Soroban
    escrowAvailable: number,  // Disponible = limit - used
    stellarBalance: number,   // Balance USDC en wallet Stellar
    pendingApprovals: number, // Monto en solicitudes pendientes
  }
}
```

### Fund Request (Driver)
```
POST /api/v1/funds/request
Body: {
  driverPubKey: string,
  managerPubKey: string,
  liters: number,
  amount: number,
  description: string
}
```

## Implementación del Contrato Soroban

El contrato debe implementar las siguientes funcionalidades:

1. **Almacenamiento persistente** de configuraciones de drivers
2. **Validación de límites** antes de aprobar solicitudes
3. **Cálculo de disponible** en tiempo real
4. **Autorización requerida** para operaciones de admin

### Consideraciones de Gas
- Usar `persistent` storage para DriverConfigs
- Minimizar lecturas/writings en cada llamada
- Considerar batching para múltiples updates
